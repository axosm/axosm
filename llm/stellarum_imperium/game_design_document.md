# 1. Overview
Stellarum Imperium is a persistent-world, browser-based MMO-RTS set in a procedurally generated universe. Players begin on a single planet in the prehistoric age and expand across solar systems and galaxies, advancing through technology eras, building empires, forging alliances, and waging war both on the ground and in the vastness of space.
The game draws on the asynchronous empire-building depth of OGame and Tribal Wars, the tile-based civilisation management of Civilization, and the grand-scale space strategy of Stellaris — all running in real time inside a web browser with no plugins required.

## 1.1  Core Design Pillars
Persistent living world — no victory condition, no resets; the universe evolves continuously.
Era progression — technology unlocks are globally time-gated so the battlefield stays balanced; players specialise, not leapfrog.
Multi-scale warfare — ground combat on planetary tiles, orbital combat above planets, and open-space combat at coordinates.
Player agency — empire-building, diplomacy, research specialisation, and military strategy all offer distinct playstyles.
Solo-friendly engineering — clean API-driven architecture designed to be built and maintained by a single developer.
# 2. Universe Structure
The game universe is procedurally generated using seeded randomness, enabling reproducible but unique galaxies, solar systems, and planets.
## 2.1  Hierarchy
Level 1 — Galaxy
The top-level container. Multiple galaxies exist. Each has a 2D coordinate space.
Level 2 — Solar System
A star with orbiting planets, asteroid belts, and empty space. Located at (x, y) in the galaxy.
Level 3 — Planet
A Goldberg-sphere world with terrain tiles. Located in a solar system at an orbital slot.
Space Objects
Asteroids, stations, and stargates also exist as coordinate-based objects in space.


## 2.2  Space Coordinates
Space (within a galaxy) is a 2D plane. Every object — ships, stations, stargates, asteroids, planets (via their solar system) — occupies an (x, y) coordinate. There are no tiles in space; movement and range are purely numerical distances.
Ships move to target coordinates over real time (seconds to weeks depending on distance and engine).
Low orbit and high orbit are distinct z-layers above a planet's coordinate.
Stargates allow fast inter-system and inter-galaxy travel. Slow travel between systems without a stargate is possible but very slow.

## 2.3  Asteroids
Asteroids are resource-bearing objects located at space coordinates. They are not colonisable as planets but players can construct mining stations on large asteroids.
Small asteroids
Cannot be built on. Ships can harvest resources directly (slow, requires presence).
Large asteroids
Support one Mining Station structure. The station extracts resources passively over time.
Ownership
Temporary — whoever controls the station or has ships present controls the asteroid. No permanent claim.

# 3. Planets
## 3.1  Goldberg Sphere Tiles
Planets are represented as Goldberg polyhedra (class I, n=0). This produces exactly 12 pentagonal tiles and a variable number of hexagonal tiles depending on the planet's size parameter.
Minimum size
~100 tiles (small moon-like body)
Maximum size
~10,000 tiles (super-Earth) — subject to balancing
Tile generation
Seeded procedural generation (similar to No Man's Sky). The seed determines tile count and terrain distribution.
Rendering
The 3D engine (Three.js) renders only the tiles visible in the current viewport for performance.

## 3.3  Building on Tiles
One building per tile. Buildings are constructed over time (seconds to days). Each building occupies the tile permanently until demolished.
Construction time depends on building type, available workers, and research bonuses.
Buildings can be damaged in combat and must be repaired over time.
The tile's terrain type determines which buildings are allowed and what bonuses apply.

## 3.4  Multi-Player Planets
Multiple players can inhabit, colonise, and contest the same planet. Territory on a planet is defined by city influence (as in Civilization) — tiles within a city's cultural radius belong to that player's empire.
Players start on their own unique home planet (no other player spawns there initially).
Colonisation ships can land on any unclaimed or contested planet.
Contested tiles trigger ground combat between occupying units.

## 4.2  Production
Resources are produced through two complementary mechanisms:
Passive generation — buildings produce resources automatically every real-time second, proportional to the building's level and terrain bonuses. No player action required.
Worker assignment — players assign population units (workers) to buildings to increase their output rate. An unassigned building still produces at its base rate.
Energy is a special resource: it is consumed by almost every building and unit. A deficit slows or halts production across the empire.

## 4.3  Storage
Each resource type has a storage cap defined by the total capacity of warehouse/silo buildings on the planet. Resources produced beyond the cap are lost. Players are incentivised to build storage or consume resources regularly.
# 5. Buildings
Buildings are constructed on planet tiles (one per tile). Each building has a type, level, current HP, and max HP. Damaged buildings produce at reduced efficiency.
## 5.1  Building Categories
Resource extraction — Mines, Farms, Lumber Yards, Oil Wells, Uranium Mines, Helium Extractors…
Storage — Warehouses, Silos, Vaults, Energy Cells…
Population — Housing, Barracks, Colonies…
Production — Foundries, Factories, Shipyards, Launch Pads…
Research — Libraries, Universities, Laboratories, Research Centers…
Defense — Walls, Towers, Bunkers, Anti-Air, Orbital Cannons, Planetary Shields…
Infrastructure — Roads (movement bonus), Spaceports, Stargate terminals…
Special — Capital (empire bonus), Command Center, Cloaking Array…

## 5.2  Construction Rules
One building per tile
A tile can never hold two buildings simultaneously.
Build time
Ranges from seconds (basic structures) to days or weeks (advanced facilities). Affected by worker count and tech.
Demolition
Takes time; returns a portion of resources.
Repair
Damaged buildings auto-repair over time if workers and resources are available. Can be accelerated manually.
Upgrade
Buildings can be upgraded to higher levels in-place. Each level increases output but also cost and time.

# 6. Units & Armies

Ground units occupy planet tiles. They are trained in Barracks or equivalent buildings and move from tile to tile over time.

HP

Cargo

Aggression

## 6.3  Ship Classes (Draft)
Scout — fast, lightly armed, high scan power.
Fighter — small, agile combat ship.
Frigate — balanced combat/defense.

Battleship — maximum firepower, slow.

📌  Ship classes are a draft. Final roster to be determined during balancing.

## 6.4  Orbit Layers
A planet has two distinct orbital layers, each with its own battle arena:
Low Orbit (LO) — close to the planet. Enables troop landing/extraction, orbital bombardment of tiles, and is accessible by planetary defense weapons.
High Orbit (HO) — further out. Safe from most planetary defenses. Ships here can intercept incoming fleets before they reach LO.
# 7. Research & Technology
## 7.1  Philosophy
Technology progression is globally time-gated to prevent power imbalances between veteran and new players. Advanced techs become researchable only after the server has been running for a defined duration. This ensures no player can access nuclear weapons while others are still fighting with iron swords.

## 7.2  Global Tech Gates
Server Age Gate
Each technology has a minimum server age (e.g. Nuclear Fission unlocks after 1 year of server runtime). No player can research it before this timer expires.
Prerequisites
Technologies also require prerequisite techs to be completed by the player (tree structure with branching choices).
Specialisation bonus
Players who invest heavily in research infrastructure can complete a tech slightly ahead of the gate (within a configurable margin).
Catch-up mechanic
New or lagging players receive an accelerated research multiplier, scaling inversely with their tech score relative to the server median.


## 7.3  Tech Tree Structure
The tree is organised by era and branches into specialisation paths. Players choose which branches to pursue, creating distinct playstyles (military, economic, diplomatic, scientific).
Each node is a technology with: name, description, cost (resources + time), prerequisites, server age gate, and effects (unlocks buildings, units, bonuses, or new resources).
Some technologies are mutually exclusive — choosing one branch may delay or block another, encouraging player identity.
The tree expands era by era: Prehistoric → Ancient → Medieval → Early Modern → Industrial → Atomic → Space Age → Interstellar.

## 7.4  Research Buildings
Research speed is determined by the number and level of research buildings on a player's planets (Libraries, Universities, Research Centers). Workers assigned to these buildings contribute to research output.
# 8. Combat
## 8.1  Combat Arenas
Planet (tile)
Ground units and defensive buildings on the same tile fight each other.
Low Orbit
Spaceships in low orbit, plus specialised planetary defense weapons (orbital cannons, shield generators).
High Orbit
Spaceship-only. Out of range of most planetary defense structures.
Open Space
Spaceships (and stations) at coordinates in the galaxy. No planetary defenses.


## 8.2  Combat Triggers
A battle begins when hostile units occupy the same location (tile, orbit, or coordinate). The trigger rules are:
Both units are flagged Aggressive → fight automatically.
One unit is Aggressive and the other is not fleeing → fight automatically.
Players are in a declared War state → all units fight automatically regardless of aggression flag.
Player manually orders an attack → fight begins regardless of flags.
Neutral / allied units never auto-fight.

## 8.3  Battle Rounds
Combat is tick-based. Each round lasts 1 real-time minute. Every round:
Each unit calculates its attack output based on its Attack stat and any active tech/building bonuses.
Damage is applied deterministically to the opposing force (no RNG). Damage is distributed across enemy units proportional to their HP.
Units whose HP reaches 0 are destroyed.
Defensive buildings on the same tile participate as stationary combatants.
The battle continues until one side is eliminated, retreats, or a ceasefire is declared.
📌  Combat is deterministic — the outcome of a battle can be predicted from unit stats alone. This allows players to plan attacks precisely.

## 8.4  Battle Reports
At the end of each round and at battle conclusion, a report is generated and pushed to involved players via SSE (Server-Sent Events). Reports include: round number, units lost per side, resources plundered (if any), and battle outcome.

## 8.5  Special Combat Mechanics
Cloaking — cloaked ships are invisible on the map unless an enemy's scan power exceeds the ship's cloak rating.
Scanning — scan actions reveal ships, troops, and building counts at a coordinate or tile.
Orbital Bombardment — ships in Low Orbit can bombard planet tiles, damaging buildings and units without a full ground invasion.
Troop Transport — ships with cargo capacity can carry ground troops across space and land them on planets.
# 9. Movement & Travel
All movement takes real time. There are no instantaneous teleports except via Stargates.
Planet tile movement
Ground units move tile by tile. Travel time scales with distance and unit speed stat.
Orbit entry/exit
Ships move between surface / LO / HO / open space over time.
Intra-system travel
Ships travel between coordinates within a solar system. Travel time depends on distance and ship speed.
Inter-system (slow)
Ships can travel between solar systems without a Stargate. Very slow (days to weeks).
Stargate (fast travel)
Players can build Stargates. Ships entering one exit at the linked Stargate nearly instantly. Stargates require significant resources and tech.
Inter-galaxy
Requires either a very long slow journey or paired intergalactic Stargates. Extremely costly.

# 10. Empires & Alliances
## 10.1  Empires
An Empire is a group of players operating as a single political entity. It is the primary unit of faction play.
Formation
Any player can found an Empire. Others apply to join; the Empire can accept or invite.
Territory
Defined by city influence rings (Civ-style). Tiles within range of a city belong to the Empire.
Capital
One planet designated as the capital. Provides bonus defense and a shared resource pool accessible to all members.
Shared resources
Members can deposit into and withdraw from the imperial treasury, subject to role permissions.
Roles
Customisable hierarchy: e.g. Emperor, General, Diplomat, Member. Roles control permissions (declare war, access treasury, manage members).
Diplomacy
Empires can declare war, sign peace treaties, form alliances with other Empires, or set neutral/hostile status.


## 10.2  Alliances
Alliances are formal agreements between two or more Empires. An alliance can include: non-aggression pacts, mutual defense treaties, resource sharing, or full military cooperation. Alliance terms are configured by the involved Empires' diplomats/leaders.

## 10.3  War
War is declared between Empires (not individual players). When at war, all units belonging to opposing Empires auto-fight on contact. War can be ended by peace treaty, which requires agreement from both Empire leaders.
## 11. Technical Architecture
## 11.1  Stack Overview
Backend
Rust with Axum web framework. Exposes a REST API and SSE endpoints.
Frontend
Vite + TypeScript. Three.js for 3D planet rendering and space view.
Desktop
Tauri wraps the web frontend into a native desktop application.
Database Dev
SQLite — local development and testing.
Database Prod
PostgreSQL — production server.
Real-time
Server-Sent Events (SSE) for battle alerts, construction completion, and other push notifications.


## 11.2  Key Design Constraints
The database schema must be compatible with both SQLite (dev) and PostgreSQL (prod). Avoid database-specific features where possible; use an ORM or query builder (e.g. sqlx with compile-time checked queries).
Planet tiles are stored as graph nodes (adjacency list) not a 2D array, since the Goldberg sphere has no regular grid.
Space objects (ships, stations, asteroids) use floating-point (x, y) coordinates. Spatial indexing (R-tree or PostGIS) recommended for prod queries.
All timed events (construction, movement, research) are stored as start_time + duration in the DB. The server computes completion by comparing to current time — no polling loop required.
Battle rounds are the only true tick: a scheduler fires every 60 seconds and resolves all active battles.

Schema Design Notes
• Use JSONB (Postgres) / TEXT+JSON parsing (SQLite) for flexible data like resource costs, effects, and cargo contents.
• Spatial queries (ships near coordinate X,Y) should use PostGIS or an R-tree index in production.
• The construction_queue and movement_orders tables replace the need for a polling game loop — ETA is computed once and stored.
• All battle state is recoverable from battles + battle_participants + battle_rounds — no in-memory-only state.
• server_config holds the authoritative server_start_time, used to compute all tech gate windows.
